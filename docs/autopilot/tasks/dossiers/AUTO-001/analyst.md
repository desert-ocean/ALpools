# Analyst Report

## Objective

Определить минимальный безопасный способ добавить шаг про подводное освещение в существующий FSM конфигуратора без переписывания сценария.

## Inputs

- пользовательский запрос;
- `AGENTS.md`;
- `docs/autopilot/sources-of-truth.md`;
- `AUTOPILOT-FOUNDATION.md`;
- `app/handlers/project_configurator.py`.

## Decisions

- Встраивать новый шаг строго после `choosing_attractions`.
- Не менять предыдущие шаги и не выносить FSM по новым модулям.
- Считать `lighting` nullable-полем FSM: `True | False | None`.
- Стоимость освещения добавлять фиксированно: `+10000 ₽`.

## Work Performed

- Проанализирован текущий порядок шагов FSM.
- Найдена точка перехода, где после аттракционов происходил прямой вызов расчёта.
- Определён минимальный write scope: один продуктовый файл.

## Self-Analysis

- Вывод: структура `design` в текущем коде фактически используется как payload по разделам, а не как отдельная доменная сущность итогового расчёта. Поэтому новые поля добавлены обратносовместимо, без рефакторинга всей модели.

## Risks

- Если в дальнейшем появится отдельный persistence/export для `design`, формат стоит централизовать в отдельной схеме.

## Handoff

Передать в `TeamLead` и затем в реализацию точечное изменение одного FSM-перехода и расчёта.
